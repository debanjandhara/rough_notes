### ğŸŸ¢ 1.2 NoSQL vs SQL Databases

### ğŸŸ¢ 3.2 Embedded Documents vs References

MongoDB allows **two ways** to relate data:

#### ğŸ“˜ Embedded Documents (Sub-documents)

When related data is stored **inside** the same document.

âœ… **Use When:**

#### ğŸ§© Reference Documents

When related data is stored in **another collection**, linked by `_id`.

**Example:**

```json

{ "_id": 1, "name": "Amit" }

// orders collection

```

âœ… **Use When:**

* Data grows large or changes independently.
* You want to avoid duplication.

#### ğŸ”„ Mongoose Example:

```js

  name: String,
  address: { city: String, pincode: Number }
```

## ğŸŸ© 7. Aggregation Framework

### ğŸ”¹ 7.1 What is Aggregation

**Definition:**
Aggregation in MongoDB is the process of transforming documents in a collection into meaningful summarized results.

**Analogy:** Like a water pipe â€” data flows from one stage to the next, getting filtered, grouped, and reshaped.

**Common use cases:**

* Calculating averages, counts, and statistics
* Joining collections (similar to SQL joins)
* Generating reports and analytics dashboards

### ğŸ”¹ 7.2 aggregate() Pipeline Syntax

The basic structure of an aggregation pipeline is:


Each stage is an **object starting with a `$` operator** like `$match`, `$group`, etc.

### ğŸ”¹ 7.3 Common Stages

| Stage      | Purpose                                               | Example                                            |
| ---------- | ----------------------------------------------------- | -------------------------------------------------- |
| `$match`   | Filters documents (like `WHERE`)                      | `{ $match: { age: { $gt: 25 } } }`                 |
| `$group`   | Groups documents by a field and performs aggregations | `{ $group: { _id: "$city", total: { $sum: 1 } } }` |
| `$project` | Shapes the output fields                              | `{ $project: { name: 1, city: 1, _id: 0 } }`       |

### ğŸ”¹ 7.4 `$lookup` (Joins)

```js

      localField: "customerId",    // field in orders
      foreignField: "_id",         // field in customers

```

## ğŸŸ© 8. Transactions & Atomic Operations

### ğŸ”¹ 8.1 What are Transactions

**Definition:**
A **transaction** in MongoDB is a sequence of read/write operations that are **executed as a single unit** â€” either **all succeed** or **none take effect**.

ğŸ‘‰ In simpler terms:

> A transaction ensures **"all-or-nothing"** behavior for your database operations.

**Example use cases:**

* Deducting money from one account and adding to another
* Updating multiple collections in sync (orders + inventory)
* Ensuring consistent data during batch updates

MongoDB supports:

* **Single-document atomic operations** (always atomic)
* **Multi-document transactions** (since MongoDB 4.0+)

---

### ğŸ”¹ 8.2 ACID Properties in MongoDB

MongoDB transactions follow the **ACID** model:

| Property            | Meaning                                  | MongoDB Implementation                  |
| ------------------- | ---------------------------------------- | --------------------------------------- |
| **A â€“ Atomicity**   | All operations succeed or none           | Rollback on failure                     |
| **C â€“ Consistency** | Data remains valid and meets constraints | Schema validation, transaction rollback |
| **I â€“ Isolation**   | Transactions donâ€™t affect each other     | Snapshot isolation                      |
| **D â€“ Durability**  | Once committed, data is permanent        | Written to disk (journaling)            |

âœ… MongoDB ensures **ACID compliance** for single and multi-document transactions (replica set & sharded clusters).

### ğŸ”¹ 8.3 Single-Document Atomicity

MongoDB guarantees **atomic operations at the document level** â€”

ğŸ‘‰ Replica Set = multiple MongoDB instances (1 primary, 1+ secondary) ensuring high availability.

**Key Notes:**

* Works in both **replica sets** and **sharded clusters** (MongoDB 4.2+)
* Slower than single operations (so use only when needed)
* You can include **multiple reads and writes** across collections

### ğŸ”¹ 8.5 Session Handling in Transactions

| Operation             | Description                                          |
| --------------------- | ---------------------------------------------------- |
| `commitTransaction()` | Finalizes all operations (data is permanently saved) |

**Transaction states:**

3. **Commit** â†’ confirm success
4. **Abort** â†’ rollback on error

### ğŸ”¹ 8.7 Error Handling in Transactions

* Network errors
* Timeouts or transient errors

**Best Practices:**

2. Use **retry logic** for transient errors.

```js
      }
      throw error;
    }
  }
}
```
